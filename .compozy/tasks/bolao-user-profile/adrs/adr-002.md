# ADR-002: View `user_predictions` para dados do perfil

## Status

Accepted

## Date

2026-06-10

## Context

A página de perfil do usuário precisa exibir predições com dados completos da partida (times, flags, grupo, horário, status, placar real) e pontos ganhos. Precisamos de uma forma eficiente de buscar esses dados.

## Decision

Criar uma nova view `user_predictions` no banco que junta `predictions`, `matches` e `profiles` em uma única query. O hook `useUserPredictions` consulta essa view diretamente.

```sql
CREATE OR REPLACE VIEW user_predictions AS
SELECT
  pr.id AS prediction_id,
  pr.user_id,
  pr.match_id,
  pr.home_score AS predicted_home,
  pr.away_score AS predicted_away,
  pr.points,
  pr.created_at,
  m.home_team,
  m.away_team,
  m.home_flag,
  m.away_flag,
  m.group_name,
  m.kickoff_at,
  m.status AS match_status,
  m.home_score AS actual_home,
  m.away_score AS actual_away
FROM predictions pr
JOIN matches m ON m.id = pr.match_id;
```

## Alternatives Considered

### Alternative 1: Query com join no hook

- **Description**: Usar `supabase.from('predictions').select('*, matches(*)')` diretamente no hook
- **Pros**: Sem migration nova, mais rápido de implementar
- **Cons**: Lógica de join fica no frontend, menos reutilizável, nomes de campos aninhados
- **Why rejected**: View centraliza lógica e simplifica o hook

### Alternative 2: Queries separadas

- **Description**: Buscar predictions, matches e profile separadamente e combinar no frontend
- **Pros**: Flexível, cada query é simples
- **Cons**: Múltiplas requisições HTTP, código de combinação complexo, mais propenso a erros
- **Why rejected**: Performance pior e mais complexidade no frontend

## Consequences

### Positive

- Single query para todos os dados do perfil
- Nomes de campos claros e consistentes
- Reutilizável para futuras features
- View herda RLS das tabelas base

### Negative

- Requer migration nova
- View precisa ser mantida se schema mudar
