# ADR-001: Implementação Completa de Notificações Push

## Status

Accepted

## Date

2026-06-11

## Context

O bolão precisa de um sistema de notificações push para aumentar o engajamento dos usuários. Foram identificados três tipos de notificação com valor distinto:

1. **Daily digest**: Resumo dos jogos do dia (08:00)
2. **Post-match**: Pontos ganhos após cada jogo finalizado
3. **Deadline reminder**: Aviso 2h antes do kickoff se o usuário não palpitou

A infraestrutura atual já possui:
- PWA configurado (installable, offline)
- Edge Function `sync-matches` com detecção de jogos finalizados
- Cron job rodando a cada 5 minutos
- Zero código de notificações existente

## Decision

Implementar **todos os três tipos de notificação em um único MVP**, incluindo:
- Infraestrutura completa de Web Push (VAPID, service worker, subscriptions)
- Sistema de opt-in (prompt inicial + configuração no perfil)
- Toggle global para ativar/desativar todas as notificações
- Os três tipos de notificação funcionais desde o início

## Alternatives Considered

### Alternative 1: Fased - Post-match primeiro

- **Description**: Implementar post-match na Fase 1, deadline na Fase 2, daily digest na Fase 3
- **Pros**: Entrega valor mais rápido, valida infraestrutura antes de escopos maiores
- **Cons**: Múltiplos ciclos de desenvolvimento, usuários recebem apenas 1 tipo inicialmente
- **Why rejected**: Post-match é emocionante mas não resolve o problema de palpites perdidos

### Alternative 2: Fased - Deadline primeiro

- **Description**: Implementar deadline na Fase 1, post-match na Fase 2, daily digest na Fase 3
- **Pros**: Foca em resolver problema imediato (esquecimento), aumenta taxa de palpites
- **Cons**: Deadline pode ser frustrante, menos "divertido" como primeira impressão
- **Why rejected**: Usuários esperam ver resultados e se divertir, não apenas lembretes

## Consequences

### Positive

- Maior impacto imediato no engajamento
- Experiência completa desde o início
- Um único ciclo de desenvolvimento (menos overhead)
- Usuários recebem valor completo desde o dia 1

### Negative

- Maior escopo inicial (mais complexidade)
- Mais tempo até primeira entrega
- Mais pontos de falha potenciais
- Testes mais extensos necessários

### Risks

- **Permissões negadas**: Usuários podem recusar notificações → Mitigação: explicar valor antes de pedir permissão
- **Entrega inconsistente**: Service workers podem ser encerrados → Mitigação: fallback para email no futuro
- **Complexidade de debugging**: Múltiplos tipos de notificação → Mitigação: logging centralizado

## Implementation Notes

- Web Push API com VAPID keys (public/private)
- Service worker customizado via vite-plugin-pwa inject
- Tabela `push_subscriptions` para armazenar subscriptions
- Edge Functions: `send-notifications` + triggers no `sync-matches`
- Cron jobs: daily digest (08:00), deadline reminders (a cada 15min)
- Post-match: trigger automático quando jogo finaliza (já detectado pelo sync-matches)
